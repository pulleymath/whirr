# Whirr — 아키텍처(개념)

PRD의 요구를 만족시키기 위한 **경계·책임·데이터 위치**만 정의한다. 구현 세부는 코드베이스를 본다.

## 신뢰·실행 경계

- **브라우저**: UI, 마이크 캡처, 로컬 세션 저장·조회, 사용자 설정 보관.
- **호스팅 앱**: 장기 비밀을 숨기고, 전사·요약 요청을 검증해 외부 서비스와 연결하는 최소 서버 역할.
- **외부 AI 서비스**: 오디오 전사와 요약 생성을 담당하며, 공급자 정책 하에 처리.

별도의 범용 백엔드(오디오 중계·비즈니스 API)는 두지 않는다.

## 녹음 후 처리 경로

기본 제품 흐름은 **녹음 후 일괄 처리**다. 브라우저는 녹음 중 오디오를 임시 구간으로 관리하고, 전사가 가능한 시점에 텍스트를 축적한다. 녹음이 끝나면 남은 전사를 마무리하고, 합쳐진 스크립트와 사용자가 입력한 노트 맥락을 바탕으로 요약을 만든다.

후처리는 같은 탭 안에서 이어지며, 실패 시 사용자가 다시 시도할 수 있어야 한다. 완료된 세션은 로컬 저장소에 남고, History와 상세 화면은 이 로컬 데이터를 읽는다.

## 구성 요소 책임

| 영역            | 책임                                                                   |
| --------------- | ---------------------------------------------------------------------- |
| 녹음 파이프라인 | 브라우저 마이크 입력을 전사 가능한 단위로 관리                         |
| 전사 계층       | 공급자 차이를 숨기고 UI에는 부분·완료·오류 상태만 전달                 |
| 요약 계층       | 스크립트와 노트 맥락을 외부 모델 요청으로 바꾸고 결과를 정규화         |
| 홈 작업면       | 노트 맥락 입력, 스크립트 확인, 요약 형식 선택, 녹음 컨트롤 접근성 유지 |
| 로컬 저장소     | 세션 단위로 제목·스크립트·요약·맥락을 보관하고 목록을 구성             |
| 앱 설정         | 브라우저 전용 설정을 저장하고 화면 전반에 제공                         |

## 데이터가 머무는 곳

- **유지**: 제목, 스크립트, 요약, 노트 맥락(클라이언트 DB).
- **클라이언트 DB(오디오)**: 전사 재시도·ZIP 다운로드를 위해 녹음 세그먼트를 IndexedDB에 보관한다. API 성공 여부와 무관하게 캡처된 구간은 먼저 저장한다.
- **서버·장기 보관 없음**: 오디오 원본은 앱 서버에 상주하지 않는다. 다운로드는 브라우저에서 생성한다.
- **서버 영속**: MVP에서 사용자 스크립트·요약 본문을 저장하지 않음.
- **브라우저만**: 전사·요약 관련 사용자 설정은 브라우저 저장소에만 두고, 비밀값은 넣지 않는다.

## 교체 가능성

STT·요약 공급자와 모델이 바뀌어도 UI는 동일한 상태 흐름만 보도록 한다. 자체 호스팅 모델로 바꿀 경우에도 제품 요구는 [PRD.md](./PRD.md)를 따른다.

## 운영·비밀

- 장기 키는 서버 환경에만 둔다.
- 공개 번들에 비밀이 섞이지 않게 한다.
- 토큰 발급 경로는 남용 완화를 위한 **속도 제한 등 정책**을 둘 수 있다(상세는 [DECISIONS.md](./DECISIONS.md)).
