# Architecture & code style review — feature-4-session-storage

## Summary

IndexedDB 계층(`whirr-db` / `sessions` / `id`·`createdAt`·`text`·`by-createdAt`)은 `docs/ARCHITECTURE.md` 3.3과 계획서와 정합적이다. `buildSessionText`를 순수 함수로 분리한 것은 규칙 공유·테스트 용이성 측면에서 적절하다. `Recorder`의 중지 흐름은 계획서의 “스냅샷은 `finalizeStreaming` 전, 저장은 이후” 권고와 일치한다. 전반적으로 범위 대비 구조가 단순하고 읽기 쉽다. 다만 **프로덕션 번들에 테스트 전용 API가 노출**되는 점과 **UI가 저장 구현체에 직접 의존**하는 점은 아키텍처·스타일 관점에서 후속 정리 여지가 있다.

## Architecture findings

| 심각도         | 항목                                                                                        | 설명                                                                                                                                                                                                                                                         |
| -------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Important**  | `resetWhirrDbForTests()`가 `db.ts`에서 export되어 프로덕션 클라이언트 번들에 포함될 수 있음 | 테스트에서 `deleteDB`·연결 해제까지 수행하는 강한 연산이 공개 API로 남는다. 트리 쉐이킹이 되더라도 심볼 노출·오용 위험이 있다. **권장**: Vitest 전용 모듈(예: `db.test-utils.ts`)로 분리하고 테스트에서만 import, 또는 빌드 타임에서 테스트 엔트리로만 묶기. |
| **Suggestion** | `Recorder` → `saveSession` 직접 의존                                                        | 계획서에서 선택 사항으로 둔 수준이지만, 세션 목록 UI·오프라인 정책이 늘어나면 “저장 포트”(함수 주입 또는 얇은 훅)로 한 겹 두는 편이 STT/녹음 UI와 저장 정책을 분리하기 좋다. 현재 이슈 범위에서는 수용 가능.                                                 |
| **Suggestion** | `getAllSessions`가 인덱스 조회 후 메모리에서 `createdAt` 내림차순 정렬                      | `getAllFromIndex` 순서에 의존하지 않고 명시적으로 정렬해 안전하다. 다만 데이터가 커지면 커서/역방향 조회 등으로 최적화 여지는 있다(현 단계에서는 과하지 않음).                                                                                               |
| (참고)         | ARCHITECTURE 3.3의 `text` 설명이 “확정된 전사”에 가깝게 적혀 있음                           | 구현은 중지 시점의 `finals`+`partial` 스냅샷으로 유실을 줄이는 선택이다. **코드 문제는 아니나**, 문서와 제품 의도를 맞추려면 PRD/ARCHITECTURE 쪽 설명을 “중지 시점까지의 화면과 동일한 텍스트” 등으로 정렬하는 편이 좋다.                                    |

## Code style findings

| 심각도         | 항목                                                                | 설명                                                                                                                                                                                                                                     |
| -------------- | ------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Suggestion** | `useCallback` 의존성 배열                                           | 계획서에는 `saveSession`(또는 래퍼) 포함을 명시했다. 모듈 상수라 런타임 버그 가능성은 낮지만, **팀 규칙이 “계획서와 동일한 deps 명시”**라면 `saveSession`, `buildSessionText`를 배열에 넣어 eslint 규칙과 문서를 맞추는 것이 일관적이다. |
| **Suggestion** | 저장 실패 시 `console.error`만                                      | 계획서 허용 범위 내. 스타일 측면에서는 나중에 토스트/인라인 메시지로 바꿀 때를 대비해 작은 `onSaveError?: (e: unknown) => void` 주입 여지를 두면 컴포넌트 테스트가 쉬워진다(필수는 아님).                                                |
| (양호)         | `WhirrDB` + `DBSchema`, 상수(`DB_NAME`, `STORE`, `DB_VERSION`) 분리 | 타입과 마이그레이션을 한곳에 모아 읽기 좋다.                                                                                                                                                                                             |
| (양호)         | `build-session-text`의 `FINAL_SEPARATOR` 상수                       | 계획서 REFACTOR 방향과 맞고, 변경점이 한 곳에 모인다.                                                                                                                                                                                    |

## Verdict

**조건부 승인(merge 가능)** — 아키텍처·스타일 기준으로 치명적 결함은 없고, IndexedDB 스키마와 녹음 중지 시 저장 흐름은 문서·계획과 잘 맞는다. **병합 전 또는 직후에 `resetWhirrDbForTests`를 프로덕션 경로에서 분리**하는 것을 권장한다. 나머지는 후속 이슈로 넘겨도 된다.
