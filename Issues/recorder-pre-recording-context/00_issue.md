# 녹음 전 회의 컨텍스트 입력

## 배경

현재 홈 녹음 흐름은 녹음이 시작된 뒤에만 회의 정보·회의록 형식 입력 영역이 펼쳐진다. 사용자는 실제로는 녹음 전부터 참석자, 주제, 키워드, 회의록 형식을 알고 있는 경우가 많다.

클로바노트의 "새 노트 생성 후 제목·참석자·메모를 미리 입력하고 원하는 시점에 녹음 시작" 흐름처럼, Whirr도 녹음 전에 회의 컨텍스트를 준비할 수 있게 바꾼다. 단, 별도 새 노트 라우트나 wizard는 만들지 않고 현재 홈의 단일 녹음 루프를 유지한다.

## 요구사항

1. 홈 idle 상태에서도 `회의 정보` 입력 카드와 `회의록 형식` 선택기를 노출한다.
2. 회의 정보 입력은 선택 사항이다. 비어 있어도 기존처럼 녹음을 시작할 수 있다.
3. 녹음 시작 버튼은 계속 즉시 접근 가능해야 한다. 컨텍스트 입력이 녹음 시작을 가로막으면 안 된다.
4. 입력한 `참석자`, `주제`, `키워드`, `회의록 형식`은 녹음 종료 후 자동 회의록 생성 enqueue payload에 반영된다.
5. 녹음 중에도 회의 정보는 계속 보인다. 뒷단 파이프라인이 바쁠 때는 기존 정책대로 입력을 비활성화하고 사유를 안내한다.
6. 세션 저장·enqueue가 성공하면 다음 녹음을 위해 회의 정보와 회의록 형식을 초기값으로 되돌린다. 저장 실패 시에는 사용자가 재시도할 수 있게 입력값을 유지한다.
7. `/recorder-preview`의 단계 라벨과 표시 조건도 새 흐름에 맞춘다.
8. `docs/USER_FLOWS.md` §2를 "녹음 전 회의 정보 입력 가능" 흐름으로 갱신한다.

## 비목표

- 화자 수 선택 UI는 추가하지 않는다. 현재 Whirr의 STT/회의록 파이프라인에 화자 수가 직접 영향을 주지 않기 때문이다.
- 북마크·메모 기능은 이 이슈에서 다루지 않는다.
- 새 노트 전용 라우트, 임시 초안 저장, 브라우저 간 동기화는 만들지 않는다.
- 마이크 권한, batch transcription, post-recording pipeline 정책은 바꾸지 않는다.

## 테스트 계획

1. idle 상태에서 `회의 정보` 카드와 `회의록 형식` 선택기가 보인다.
2. 회의 정보가 비어 있어도 `시작` 버튼으로 녹음 시작이 가능하다.
3. 녹음 전에 입력한 참석자·주제·키워드가 녹음 종료 후 enqueue payload의 `sessionContext`에 들어간다.
4. 녹음 전에 선택한 회의록 형식이 enqueue payload의 `meetingTemplate`에 들어간다.
5. 세션 저장·enqueue 성공 후 입력값이 초기화된다.
6. 저장 실패 시 입력값이 유지되고 오류 문구가 보인다.
